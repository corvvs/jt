# 技術メモ

どう考えても README に書くものではないのでそのうちファイル名が変わります。

## 行アイテム

1 つの JSON が持っているキーパス 1 つが行アイテム 1 つになる。

次の JSON を例にとる:

```json
{
  "array": ["abc", "d"],
  "object": {
    "subarray": [1],
    "str": "ing"
  }
}
```

この JSON は,

- `array`
- `array.0`
- `array.1`
- `object`
- `object.subarray`
- `object.subarray.0`
- `object.str`

以上 7 個のキーパスを持つ。\
これにルート(キーパス``)に対応するものを加えて, 合計で 8 個の行アイテムが生成される。

## 絞り込み階層構造

入力 JSON から得られる行アイテムは, 次のリストの順に絞り込みを施され, 最終的に表示される行アイテムが定まる:

1. ナローイングによる絞り込み
2. 検索クエリによる絞り込み
3. フォールディングによる絞り込み

## 各種操作の対象

- 全フォールド(Fold All)ボタン
- 全アンフォールド(Unfold All)ボタン
  - どちらも「検索クエリによる絞り込み」の後、「フォールディングによる絞り込み」の前。

## 検索クエリ言語 (Advanced)

実装は `libs/advanced_query/` (tokenizer / parser / matcher)。パーサは再帰下降。
クエリは行アイテム 1 つずつに対して真偽を返す (matcher はアイテム単位の純関数)。

### 構文

```
query     := orExpr
orExpr    := andExpr ("|" andExpr)*        ... & は | より強く結合する
andExpr   := unit ("&" unit)*
unit      := "(" orExpr ")" | simple
simple    := keypath (":" value)? | ":" value
keypath   := "$"? "."? segment ("." segment)*   ... 先頭の . は任意
segment   := (key | "**") predicate*
predicate := "[" simple "]"
```

- **キーパス**: `.` 区切り。セグメントには `*` (任意文字列のワイルドカード, `sta*us` のような部分使用も可) と
  `**` (0 個以上の任意セグメント) が使える。照合は**後方一致** — `$` を前置するとルート直下からの完全一致になる。
- **値**: `:` の後ろ。裸の値は空白と `& | ) ]` 以外なら何でも含められる (`1.5`, `12:30` など)。
  空白や記号を含む文字列は引用形 `"..."` で書く (`\"` と `\\` のエスケープあり)。
- **述語 `[...]`**: セグメントが照合したノードの**部分木**が満たすべき条件。
  中身は 1 本の keypath(:value)。複数並べると AND。ネスト可。

### 意味論

- キーパスと値の条件は**同じノード (チェーン末端)** で評価される。
  `items.*.status:"error"` にマッチするのは値が `"error"` である status ノードだけ。
- 述語の中は「述語を付けたノードをローカルルートとする世界」:
  指定なしなら部分木の任意の深さ、`$` 前置なら直下からの完全一致。
  値のみの述語 `[:v]` は「部分木のどこか (自身を含む) に値 v がある」。
- 値の比較は 2 種類:
  - 引用形 → **文字列値との厳密一致** (`count:"42"` は数値 42 にマッチしない)
  - 裸 → toString 比較 (型を見ない。数値・真偽値は文字列表現で比較)。`:null` は JSON null にマッチ

### 例

```
items.*.status:"error"            値が "error" の status ノード
items.*[status:"error"]           status が "error" の items 要素 (祖先にマッチ)
users.*[status:"active"].posts.*  active なユーザーの posts 要素 (条件ノードの配下にマッチ)
$.items.**.id                     ルート直下 items の配下, 任意の深さの id
meta.price:1.5 | meta.tag:"a b"   小数と空白入り文字列
req[a:1][b:2]                     部分木に a=1 と b=2 の両方を持つ req
```

### 制限 (v1)

- 述語の中に `& | ( )` は書けない (述語を並べた AND で代用)
- `**` に述語は付けられない
- キー名に `. : $ ( ) [ ] & |` や空白を含む場合、そのキーは検索できない (キー側に引用構文が無い)

## ファセット絞り込み (プロファイル → 検索)

プロファイルパネルの値分布の行, および true/false・null の統計チップをクリックすると,
そのキーパスパターンと値への等値クエリを合成して advanced 検索に流し込む。
合成は `libs/facet.ts` の純関数。

仕様上の取り決め:

- 合成形は `$.<キーパスパターン>:<値>`。文字列は引用形 (厳密一致), 数値・真偽値・null は裸。
  カウントが 0 の true/false チップはクリック不可。
  ナローイング中は rootKey が前置されるため, クエリは自然にそのサブツリーへ限定される
- クリック時に検索モードを advanced に切り替え (既定モードの保存も通常のモード切替と同じ扱い),
  検索パネルを開く。もう一度クリックするとクエリを消して解除する。
  現在のクエリが合成クエリと一致している行は絞り込み中の表示になる
- 表現できないものはクリック不可 + title で理由を示す:
  キー名にクエリ言語の構造文字 (`. : $ ( ) [ ] & |`) や空白・`*` を含む場合,
  値に制御文字を含む場合 (input で編集できない形になるため)
- resultAppearance はユーザー設定のまま触らない

## ピン

行アイテムにピン (+メモ) を打ち, ドキュメントに紐づけて IndexedDB に保存する。
コーデック相当は `libs/pins.ts`, 永続化は `data/pins.ts`, 状態は `states/pins.ts`。

仕様上の取り決め:

- ピンの同定は行 index ではなく**キーパス (`JsonRowItem.elementKey`)** で行う。
  ドキュメントの編集で index はずれるが, キーパスならオブジェクトキーの増減に対して安定する。
  解決できなくなったピンは一覧の末尾に警告付きで残る (勝手に消さない)。
- 保存先は `DocumentPins` ストア (IndexedDB schema v2 で追加, id = doc_id)。
  数MBになりうるドキュメント本文を書き直さずにピンだけ更新するため, 本体とは別ストア。
  ドキュメント削除時にピンも一緒に消す。
- 保存済みピンの読み込みが完了するまでピンは打てない (`loaded` フラグ)。
  読み込み前の書き込みを許すと, 遅れて届いた読み込み結果と新しいピンが相互に上書きし合うため。
- 行のピングリフはボタンで, 押すとグリフ直下にメモバルーンが開閉する。
  バルーンの初手は読み取り専用の表示で, メモ本文は選択・コピーできるプレーンテキスト。
  編集への切替はバルーン内・パネル内とも鉛筆アイコン (本文クリックでは編集にならない)。
  ピンを打った直後だけは編集状態 (空の下書き) で開く。
  編集は Enter・グリフ再クリック・フォーカス喪失で確定 / Esc で破棄。
  何も入力しなければメモ無しのピンのまま閉じるだけで, メモ不要時の追加アクションは無い。
- メモ付きのピンを外してもメモは即座には失われない。メモはドキュメント単位で退避され,
  同じキーパスにピンを付け直すと復元される (バルーンが復元済みメモ入りで開く)。
  退避はセッション内の安全網で永続化はせず, ドキュメント切替でクリアされる。
- バルーンが開いている間は行のメニューを出したままにする (ピンボタンの位置が動かない)。
  この状態で既にピンは立っており, もう一度ピンボタンを押すと下書きごとピンが外れる (2状態トグル)。
  バルーンが開いている行のピンボタンとグリフは mousedown で preventDefault し
  入力からフォーカスを奪わない — blur (確定) が先に走ると再レンダリングで
  クリックが失われるため。
- ピンへのジャンプは (1) 祖先のフォールドを開き (2) 対象を含む所までナローイングを pop してから
  スクロールする。検索の絞り込みで隠れている場合はジャンプせず toast で知らせる。
- バルーンの表示状態から前後のピンへジャンプできる (↑/↓ ボタンまたは矢印キー。文書順・端で折り返し)。
  移動先でもバルーンが表示状態で開くので, ピンとメモを順に巡回できる。
  キーパス未解決のピンは巡回からスキップされる。
- diff モードはピンの対象外 (行の index 空間が新旧のマージ列になるため)。
- 未保存の一時ドキュメント (docId = "new") にはピンを打てない。
- パネルの開閉は Cmd/Ctrl+Shift+P またはヘッダーの Pins ボタン。

## 共有リンク

ドキュメント本文とビュー状態(ナローイング・クエリ・フォールド)を gzip + base64url で
URL フラグメントに詰めたもの。形式は `/shared#j1.<base64url(gzip(JSON))>`。
フラグメントはブラウザからサーバに送信されないため、データはクライアントの外に出ない。
コーデックは `libs/share.ts`、受信フローは `components/Main.tsx` の shared 分岐。

仕様上の取り決め:

- 受信時の docId は本文のハッシュから決定論的に導出する(`shared-<sha256の先頭32hex>`)。
  同じリンクを何度開いても同じドキュメントに落ち、重複保存されない。
- ただし受信者がそのドキュメントをローカル編集済みの場合は上書きせず、別ドキュメントとして新規保存する。
- ペイロードの `format`(json/jsonl)が受信者のグローバル設定と異なる場合、設定を切り替える
  (ファイル D&D と同じ挙動)。
- diff モードのビュー共有は対象外(diff 中は Share ボタン無効)。
